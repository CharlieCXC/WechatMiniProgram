## 0. TL;DR

**搜个仙儿** 是一个邀请制的传统文化咨询撮合平台，**核心命题**是用结构化评价替代口口相传，把"找熟人推荐师傅"这件事数字化、可信化。

**核心定位 framing（合规生命线）**：**咨询撮合 ≠ 测试工具**。本平台是 marketplace（类比在行/好大夫/简单心理），不是算法生成结果的工具型应用。

**架构原则**：**能用微信标准接口的，尽量使用微信标准接口**（登录 / 支付 / 订阅消息 / 内容审核 / 媒体上传 等优先用 `wx.*` 官方 API；非腾讯生态能力以腾讯云为优先备选）。

**MVP 覆盖**：
- C 端微信小程序（5-tab：求问 / 师傅 / 消息 / 订单 / 我的）
- 师傅 H5 后台（Ant Design Pro）
- 异步报告类 + 实时 IM 咨询（文字/语音/图）
- 邀请制 + 分段担保 + 仲裁机制 + 评价飞轮

**关键约束**：
- 单一主体：新建专用公司，跨主体风险隔离
- 类目：生活服务 → 信息咨询服务（主）
- 抽佣 10%（卖方定价市场）
- 不做实时音视频 / 线下服务 / AI 算命 / UGC 社区（V2+）

**关键路径**：公司注册 → 域名 ICP 备案 → 小程序 + 商户号 → 开发 → 首审 → 上线
**预计上线**：开发启动后 4 个月（含 ICP 备案瓶颈）

## 1. 产品定位与核心 thesis

### 1.1 一句话定位

> 一个邀请制的传统文化咨询撮合平台，用结构化评价替代口口相传，把"找熟人推荐师傅"这件事数字化、可信化。

### 1.2 Slogan

> **搜了个仙儿，功德加一**

### 1.3 产品 thesis

**真问题**：用户有传统文化咨询需求时，传统靠熟人介绍，但圈子有限、信息不对称、踩雷无追溯。

**真壁垒**：信任。靠两件事建立：
1. **邀请制 + 创始人亲选** → 供给端从第一天就是精筛
2. **结构化评价 + 闭环交易** → "口碑"变成可累积、可比、可追溯的资产

**真护城河**：不是工具型小程序（任何人都能做"在线塔罗"），而是「**严选社区 × 评价飞轮**」——大平台学不来，因为他们不可能像创始团队这样一对一访谈每位师傅。

### 1.4 类比对象

| 类比对象 | 共同点 | 关键差异 |
|---|---|---|
| 在行（付费咨询撮合） | 1对1 真人咨询 + 平台撮合 | 我们更垂直传统文化领域 |
| 好大夫在线 | 严选 + 真人服务 + 评价飞轮 | 我们门槛更轻 |
| 简单心理（心理咨询撮合） | 私密咨询场景 + 平台担保 | 我们文化属性更强 |

**反对类比（必须撇清）**：朱雀算命 / 易奇八字 / 星座运势 / 抽签等"算法生成结果"的测试工具型应用——这些是微信小程序明令禁止的"测试类内容"，我们不属于此类。

## 2. MVP 范围与边界

### 2.1 在范围内

| 模块 | 说明 |
|---|---|
| 用户 C 端小程序 | 浏览 / 求问入口 / 师傅 feed / 售前 IM / 下单 / 服务消费 / 评价 |
| 师傅 H5 后台 | profile 管理 / 接单 / 写报告 / 案例管理 / 财务收益 / 预约时段 |
| 师傅端小程序模块 | 接单通知 / IM 沟通 / 订单状态查看（轻量版） |
| 订单系统 | 异步报告类 + 实时 IM 咨询类，状态机驱动 |
| IM 系统 | 文字 + 语音消息 + 图片（基于腾讯云 IM SDK） |
| 评价系统 | 结构化维度评分 + 预设标签 + 自由文字 + 低分公开展示 + 师傅一次回应 |
| 担保 & 仲裁 | 分段担保规则 + 平台人工裁决（创始团队前期亲自处理） |
| 支付系统 | 微信支付 + 平台代收代付 + 抽佣 10% |
| 师傅入驻流程 | 邀请制 + 视频访谈 + AI 转写抽取 + 创始人润色 + 师傅签字 |
| 合规护栏 | 文案规避词清单 + 师傅入驻培训 + IM 输出关键词风控 |

### 2.2 不在 MVP 范围

V2+ 评估：
- 实时音视频通话（TRTC）
- 抖音小程序版本（Path C）
- 师傅短视频内容流
- 用户优惠券系统进阶（节日券/老带新券等）
- 用户案例分享区

V3+ 评估：
- 线下服务（风水/法事/上门）+ GIS 定位
- 师傅等级体系（青铜/白银/金牌）
- 用户 VIP 会员
- 平台编辑测评专题

V4+ 评估：
- 师傅培训学院
- AI 售前助手（合规备案后）
- 内容社区 / UGC
- 异业合作 / 海外华人市场

明确不做：
- 任何算法自动生成结果的"测算工具"（如"输入八字→系统出报告"）
- 售前 AI 智能体（合规风险高）
- 任何"改运""化解""法事""驱邪"类服务

## 3. 信息架构与用户旅程

### 3.1 底部 5-tab 结构

```
┌──────┬──────┬──────┬──────┬──────┐
│ 求问  │ 师傅  │ 消息  │ 订单  │ 我的  │
└──────┴──────┴──────┴──────┴──────┘
```

### 3.2 Tab 详细设计

**Tab 1「求问」** — 事项导向，服务新手 + 体现"平台撮合"价值
- 首页事项卡片：感情咨询 / 事业咨询 / 财运分析 / 健康咨询 / 求子咨询 / 决策探讨 / 起名 / 流年 ...
- 点击事项进入：「想问 XX 的话，可用 X/Y 方式，推荐这几位师傅」
- **这一 tab 体现"平台撮合"价值，不是目录**

**Tab 2「师傅」** — 师傅 feed，服务回头用户 + 信任飞轮
- 大众点评式卡片流，每张卡：头像 + 简介 + 严选徽章 + 擅长方式 + 擅长事项 + 评分 + 服务人次 + 最近优质评价摘录 + 起价
- 筛选：方式 / 事项 / 价格区间 / 严选认证
- 排序：综合 / 评分 / 最近评价 / 服务人次

**Tab 3「消息」** — IM 入口
- 用户与各位师傅的对话列表
- 按最近活跃排序
- 未读红点
- 与「订单」tab 通过 deep link 互跳

**Tab 4「订单」** — 订单管理
- 进行中（待支付 / 待响应 / 处理中 / 待接收 / 待评价）
- 已完成 / 已取消
- 订单卡片可跳转到对话或交付物详情

**Tab 5「我的」**
- 头像 + 昵称 + 钱包 + 收藏师傅 + 历史咨询 + 设置 + 客服 + 关于「搜个仙儿」

### 3.3 三条核心用户旅程

**Journey A — 新用户事项驱动（首单转化主路径）**
```
打开 → Tab1「求问」→ 选「感情咨询」→ 看 5 位擅长感情的师傅
  → 进师傅 profile（自述/案例/评价/徽章）
  → 「向师傅咨询」→ 售前 IM（5 条免费额度）
  → 师傅响应 + 协商时段 → 选 SKU + 预约 → 支付
  → 服务交付 → 用户确认 → 评价
```

**Journey B — 回头用户师傅驱动（复购路径）**
```
Tab2「师傅」feed → 浏览/筛选 → 师傅 profile → 同 Journey A 从第 3 步
```

**Journey C — 师傅接单（跨 H5 + 小程序）**
```
推送/通知 → 看留言/预约请求 → 回复 + 确认时段
  → 用户支付 → "待交付"
  ├── 异步: H5 写报告上传 artifact
  └── 实时: 小程序到点 IM 沟通
  → 交付完成 → 用户确认/超时确认 → T+7 进结算池
```

## 4. 状态机

### 4.1 异步报告类订单

```
[已下单] ──师傅接单 ≤24h──→ [处理中] ──交付 artifact──→ [已交付]
   │ 超时                                                  │
   └──→ [自动取消·全额退]                                   │
                                       ┌──────────────────┼──────────────────┐
                                       ↓                  ↓                  ↓
                                   用户确认            7天超时           用户异议
                                   [已完成]            [已完成]          [仲裁中]
                                                                            │
                                                              ┌─────────────┴─────────────┐
                                                              ↓                           ↓
                                                          裁决退款                    裁决正常
                                                          [已退款]                    [已完成]
```

异步类强制要求 **artifact**：师傅必须上传具体交付物（文字报告 + 语音 ≥5 分钟 + 卦象/牌阵图），不能只在 IM 说"批好了"。**这是事后仲裁的关键证据**。

### 4.2 实时 IM 咨询类订单

```
[已预约] ──双方点头──→ [待支付] ──支付──→ [已支付] ──到点──→ [咨询中]
                                                                  │
                                          师傅失联 ──→ [全额退]
                                                                  │
                                          时长到 / 双方结束 ──→ [咨询结束]
                                                                  │
                                          ┌───────────────────────┼──────────────┐
                                          ↓                       ↓              ↓
                                      用户确认               24h 超时         异议
                                      [已完成]               [已完成]         [仲裁中]
```

实时类强制留痕：所有 IM 在平台内（closed-loop 闭环的硬性要求）。文字明文 + 语音/图片加密存储 30 天用于仲裁。

### 4.3 担保规则映射

| 触发场景 | 系统行为 | 退款规则 |
|---|---|---|
| 师傅 24h 不接单 | 自动取消 | 全额退 |
| 超时未交付 | 用户一键申退 | 全额退 |
| 用户发起"敷衍异议"（交付物简陋/IM 长时间无回应） | 平台仲裁 | 部分/全额退 |
| 用户觉得"不准" | — | 不退，可低分评价 + 文字吐槽公示 |

**沉默规则**：用户收到报告 7 天不操作 = 默认完结，钱 release 给 师傅（否则 师傅 永远拿不到钱）。

## 5. 核心模块

### 5.1 师傅系统

#### 5.1.1 邀请 & 入驻流程

```
邀请 (创始人私域发出带 inviter_id 的链接)
  → 师傅 微信扫码进入 H5 后台
  → 实名认证（身份证 + 手持照，腾讯云人脸核身）
  → 提交基础信息（从业年限 / 师承 / 擅长方式 / 事项标签）
  → 预约 30 min 视频访谈
  → 创始人亲自访谈（系统录音）
  → 后台 AI 转写 + 字段抽取 → profile 初稿
  → 创始人润色 → 师傅 在线签字确认
  → 终审通过 → 上架 + 工具教学
```

#### 5.1.2 师傅 Profile 数据模型

| 字段类 | 内容 |
|---|---|
| 基本 | 师傅名号 / 头像 / 简介（≤50 字） |
| 内容 | 师承经历 / 解读理念 / 自述视频（≤3min） |
| 标签 | 擅长方式（多选）/ 擅长事项（多选） |
| 案例库 | N 个脱敏案例（文字 + 可选图片，由订单沉淀） |
| 服务 | M 个服务 SKU |
| 排期 | 周历式可预约时段 |
| 徽章 | 严选徽章（创始人手动授予） |
| 量化（自动） | 评分 / 服务人次 / 复购率 / 平均响应速度 / 准时率 |

#### 5.1.3 师傅 H5 后台模块

| 模块 | 核心功能 |
|---|---|
| 仪表盘 | 今日待办（未接单 / 未交付 / 异议）+ 收益概览 |
| 订单管理 | 接单 / 拒单 / 改约 / **改价** / 申请延期 / 上传交付物 / 异议申诉 |
| Profile 编辑 | 自述 / 案例 / SKU 管理 / 可预约时段 |
| 案例管理 | 引导从已完成订单沉淀脱敏案例（含用户授权流程） |
| 财务 | 待结算 / 已结算 / 提现 / 历史明细 |
| 运营沟通 | 联系平台，上传申诉证据等 |

### 5.2 订单系统

#### 5.2.1 服务 SKU 模型

师傅自定义 SKU：
- 名称（如「八字解读咨询」「30 分钟塔罗答疑」）
- 类型：**异步报告** / **实时 IM 咨询**
- 价格（师傅自定，平台不限上下限，最低 ¥99 引导价）
- 异步类：承诺交付时长（24h / 48h / 72h）
- 实时类：单次时长（30 / 60 min）
- 简介 + 包含内容描述

#### 5.2.2 订单关键字段

```
order_id / user_id / master_id / sku_id / sku_snapshot
state / created_at / accepted_at / delivered_at / completed_at
original_price / price_change_history / final_price / platform_commission
关联：conversation_id / artifact_ids[] / dispute_id?
```

#### 5.2.3 关键事务

| 角色 | 动作 | 触发状态 |
|---|---|---|
| 师傅 | 接单 / 拒单 | 已下单 |
| 师傅 | 改约时段 | 已下单 / 已接单 |
| 师傅 | 改价（需用户确认） | 已下单 / 已接单 / 处理中 |
| 师傅 | 申请延期（需用户确认） | 处理中 |
| 师傅 | 上传交付物 + 交付 | 处理中 |
| 师傅 | 异议申诉 | 仲裁中 |
| 用户 | 确认收货 | 已交付 |
| 用户 | 发起异议 | 已交付 / 咨询结束（7 天内） |
| 用户 | 评价 | 已完成 |
| 用户 | 申请退款（按分段规则） | 已下单 / 已接单 / 处理中 |

#### 5.2.4 改价流程

```
师傅 H5 后台 → 订单详情 →「调整价格」按钮
  → 输入新价 + 调整原因 → 发起改价请求
  → 用户对话框收到系统卡片:「师傅 申请调价至 ¥399（原 ¥199），原因：xxx」
  → 用户：接受 / 拒绝 / 取消订单
  → 接受 → 价差补付/退款 → 新价生效
  → 拒绝 → 师傅 可再发一次；或用户取消（按状态走分段退款）
```

允许改价状态：[已下单 / 已接单 / 处理中]；[已交付 / 已完成] 后锁死。

### 5.3 IM 系统（BOSS直聘式单对话框）

#### 5.3.1 数据模型

- **conversation**：1 用户 × 1 师傅 = 1 个，终身唯一
- **message**：sender / type（文本/语音/图片/system_card）/ content / created_at / order_id?
- **system_card 类型**：订单创建 / 改价请求 / 改约请求 / 延期申请 / 订单完成 / 评价提交

#### 5.3.2 限额逻辑

```
用户发消息时:
  if 已发未回数 >= 5 AND 师傅 从未响应过 AND 该会话无已建订单:
    拒绝发送, 提示「请等师傅回应，或先发起预约下单」
  else:
    放行 (师傅 响应任一条 → 已发未回数清零, 解锁)
```

师傅 响应 ≥1 条立即解锁，之后无限制。下单后完全无限制。

#### 5.3.3 风控

- 发送前：关键词词典过滤（红线词 + 敏感词）
- 图片：OCR + 内容安全 API（腾讯云）
- 24h 师傅未回 → 推送提示用户「师傅响应慢，要不要看看其他师傅」

### 5.4 评价系统

#### 5.4.1 量化维度（4 项 1-5 星）

- **解读专业度**
- **沟通耐心**
- **仪式感**
- **性价比**

**刻意不设"准不准"维度**——"准"既迷信化又主观，也容易触发审核。

#### 5.4.2 标签

预设：耐心 / 有共鸣 / 仪式感强 / 指引清晰 / 准时 / 用心 / 文笔好 / 解读深入

自定义标签需平台审核后入预设池。

#### 5.4.3 文字评价 + 师傅一次回应权

- 评价提交后 师傅 收到通知，**7 天内可公开回应一次**（永久公开，不可改）
- 评价提交 24h 后才生效（冷却期，可撤回）

#### 5.4.4 展示策略

- 默认展示全部（含低分）
- 排序：最新 / 最高 / 最低 可切换
- 师傅主页"评价"tab 展示全部
- 师傅卡片摘要：最近 1 条高分评价摘录（≤30 字）

### 5.5 仲裁系统

#### 5.5.1 异议发起

- 入口：订单详情页（已交付 / 咨询结束 后 7 天内）
- 理由模板：师傅敷衍 / 失联 / 交付物简陋 / 服务不符 SKU 描述 / 其他
- 用户附文字说明 + 截图证据

#### 5.5.2 仲裁流程

```
用户发起异议 → 平台运营（创始人）收到通知
  → 24-48h 内介入: 查看 IM 对话 + 交付物 + 订单记录
  → 必要时联系双方 5 min 陈述
  → 裁决: 全额退 / 部分退 / 驳回
  → 双方收到裁决书 + 理由说明
```

#### 5.5.3 透明度建设

每月公示「申诉裁决数据」（异议率 / 各类裁决比例 / 平均处理时长），程序正义可见。

### 5.6 支付结算系统

#### 5.6.1 资金流

```
用户付款 → 微信支付到平台商户号 → 进入"待交付资金池"
  → 订单完成
  → 资金分配: 师傅 90% 进"待结算" + 平台 10% 抽佣
  → T+7（评价窗口结束）→ 师傅 可提现余额
  → 师傅 H5 发起提现 → T+1 到账 微信零钱 / 银行卡
```

#### 5.6.2 异常资金流

- 异议中：资金冻结池中，直至裁决
- 退款：直接从池中退给用户，不经过 师傅 已结算余额

#### 5.6.3 抽佣

- 平台抽佣 **10%**（远低于行业 20-30%）
- 抽佣从师傅收入扣，用户视角看到的是原价（不显示"服务费"add-on）
- V3+ 视业务量调整（10% → 12% → 15%）

## 7. 合规策略

### 7.1 核心 framing（贯穿全产品）

**「搜个仙儿是咨询撮合平台，不是算命工具」**

| 服务交付 | 自动测算报告 | 真人撰写的咨询解读 |

## 9. 技术栈

### 9.0 架构原则（最高优先级）

> **能用微信标准接口的，尽量使用微信标准接口。**

具体表现：

| 能力 | 优先选择（微信标准） | 备选 / 互补 |
|---|---|---|
| 用户登录 | `wx.login` + `auth.code2Session` | — |
| 支付 | 微信支付 JSAPI（`wx.requestPayment`） | — |
| 推送通知 | **微信订阅消息** (`wx.requestSubscribeMessage`) | — |
| 文本审核 | **`msgSecCheck`**（云调用） | 腾讯云内容安全（后审备份） |
| 图片审核 | **`imgSecCheck`**（云调用） | 腾讯云内容安全（后审备份） |
| 媒体上传 | `wx.uploadFile` + `wx.chooseMedia` | — |
| 录音 | `wx.startRecord` / `wx.stopRecord` | — |
| 图片预览 | `wx.previewImage` | — |
| 分享 | `wx.shareAppMessage` | — |
| 二维码 | `wx.scanCode` / 小程序码（`getUnlimited`）| — |
| 卡券 / 会员 | （V2+ 评估）微信卡包 | — |

**用微信标准接口的三重好处**：
1. **审核更友好** — 用官方能力降低被认定"功能可疑"的概率
2. **体验更原生** — 用户对授权/弹窗等交互已有心智模型
3. **维护更省心** — 微信侧升级时无须自己跟进

**仍需腾讯云生态（非微信原生）的能力**：
- 用户↔师傅 IM（小程序无原生用户间 IM，**腾讯云 IM TIM** 是腾讯生态内最匹配的选择）
- 对象存储（**腾讯云 COS**，与小程序生态契合）
- 语音转文字（**腾讯云 ASR**，仅用于离线访谈处理）
- 人脸核身（**腾讯云人脸核身**，比小程序自带能力更通用）

**不用云开发（CloudBase）的原因**：业务复杂度需要关系型数据库 + 后端 + H5 后台一致架构，云开发偏小程序内嵌场景，不适合双端通用。

## 12. 冷启动策略

### 12.1 起点盘点（坦诚）

- 公众号 / 抖音 / 视频号：❌ 无
- 已有用户：❌ 无
- 师傅 资源：⚠️ 太太圈子可能有
- 资金：⚠️ 个人项目，预算有限
- 时间：兼职投入

**结论**：双边市场先攻克**供给端（师傅）**——有师傅才有用户来"求问"。

### 12.2 供给端冷启动（师傅）— Month 1-3

**主路径：创始人人脉网络**
- 太太熟悉这个圈子 → 直接邀请（30 分钟视频访谈 + 即时上架）
- 目标：前 30 位 师傅 全靠人脉

**辅助路径：玄学博主邀请**
- 知乎 / 小红书上有粉丝的塔罗师、八字师、命理师
- 邀请话术核心：
  > "10% 抽佣（行业 25%）、严选品牌、邀请制冷启动期上架免审、长期 founder 直接对接"
- 目标：通过博主带来 20+ 师傅

**激励机制**：
- "老带新"奖励：师傅 推荐另一位 师傅 入驻并完成首单 → 老 师傅 获得 ¥200 奖金 + 1 个月免抽佣
- 首批严选标识：前 50 位 师傅 自动获得"创始严选"徽章（永久）

**KPI**：Month 3 末 ≥ 50 位上架师傅，覆盖 八字 / 塔罗 / 六爻 / 紫微 / 起名 5 大方式

### 12.3 需求端冷启动（用户）— Month 3-6

**主路径：师傅 私域裂变**（最关键）
- 每位 师傅 入驻后获得专属推广海报 + 二维码
- 师傅 在自己朋友圈 / 原有客户群发布
- 用户通过 师傅 二维码进入小程序，自动绑定该 师傅，首单优惠 ¥30
- 师傅 通过自己客户的单子获得收益，有动力推广

**辅助路径 1：建公众号（仅做内容引流，不做交易）**
- 注册一个独立公众号（主体可以是创始人个人，与小程序公司主体隔离）
- 内容矩阵：「如何在搜个仙儿找到适合自己的咨询师」「八字咨询的 N 个常见误区」等
- 定位正向化教育内容
- 公众号底部菜单挂小程序入口

**辅助路径 2：小红书 + 知乎种草**
- 创始人兼职做内容（每周 1-2 篇）
- 小红书：分享真实咨询案例（脱敏）、"如何选靠谱师傅"等
- 知乎：回答"塔罗师靠谱吗" "如何找到好的八字师傅"等
- 软性引流到公众号 → 公众号引流到小程序

**辅助路径 3：种子用户活动**
- "首单 9.9 元体验"（覆盖 5 分钟 IM 答疑类 SKU）
- "邀请好友首单各得 ¥50 券"
- 在创始团队朋友圈 / 微信群投放

**KPI**：Month 6 末 月活 ≥ 500 / 累计订单 ≥ 200 / 评价数 ≥ 100
